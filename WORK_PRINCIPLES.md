---

## 三条核心原则

### 原则 1：项目文档化
> 探索项目代码 → 生成功能描述文档 → 每次修改都参考并更新

**执行方式**：
- 接手任何项目时，先探索代码结构
- 生成 `FUNCTIONS.md` 或类似的功能描述文档
- 每次修改代码前，必须先阅读该文档
- 修改完成后，必须同步更新文档

**目的**：保持代码和文档同步，避免"代码改了但没人知道"

---

### 原则 2：所有项目统一执行
> 以后所有项目，都要按照原则 1 操作

**执行方式**：
- 不管是什么项目（webviewer、新技能、脚本等）
- 都要创建功能描述文档
- 都要遵循"修改前读文档，修改后更新文档"的流程

**目的**：建立统一的工作标准，保证所有项目都可维护

---

### 原则 3：任务拆解执行
> 以后所有项目修改，都要把要求拆分成 tasklist，并逐个执行

**执行方式**：
- 收到任何修改需求时，先拆解成具体的 task list
- 按顺序逐个执行每个 task
- 每个 task 完成后打勾 ✅
- 全部完成后汇报结果

**示例**：
```
Task List:
1. [ ] 探索项目结构
2. [ ] 分析核心功能
3. [ ] 生成功能文档
4. [ ] 验证文档完整性
```

**目的**：避免遗漏，保证工作质量，让用户清楚进度

---

## 违反后果

如果不遵守这些原则：
- 代码和文档会脱节
- 后续维护成本飙升
- 九魁会不爽 😠
